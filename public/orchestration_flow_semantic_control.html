System Architecture Diagram

编排流程：语义控制如何转化为多轮设计动作

当 semantic layer 已经成形之后，agent 的角色不再是直接生成，而是接管多轮判断、策略切换与方向收束。
这不是一次性生成流水线，而是一个以设计状态为核心对象的循环系统：用户反馈、已选方向、品牌线索与前轮状态会先被解释为可比较的语义状态，再由 agent 决定下一轮应该如何推进、试探、收束或切换。
状态层为核心控制对象
编排层负责策略决策
执行层负责受控输出
模块 01 系统输入

输入层

系统在每一轮开始前读取的信号集合
Input Signals
A
用户反馈
“更空一点 / 质感别丢 / 更像地毯一点”
B
当前已选方向
用户当下偏好的图像、结构方向或语义路线
C
品牌线索库
从历史设计中反向读取出的品牌语义 cues
D
前轮状态
前轮选择、排斥项、已锁定倾向与继承约束
系统并不是从空白 prompt 重新开始。每一轮都从语言、选择、品牌线索与前轮状态共同起步。
模块 02 语义解释

语义解释层

将当前回合的混合信号转译为可比较的设计状态
Interpretation Layer
01
意图识别
用户真正要改的是主题、材料、密度、构图，还是“更像地毯”的整体语法。
02
槽位映射
将变化落到 motif / arrangement / density / surface / material / cue 对齐等可控槽位。
03
状态更新
识别哪些倾向被加强、削弱，哪些变量是本轮新引入的。
04
线索匹配
判断当前方向真正对齐了哪些品牌语义 cues，而不是机械继承全部历史线索。
这一层的职责不是生成结果，而是把模糊反馈转成结构化状态，为后续决策提供统一对象。
模块 03 核心状态对象

当前设计状态

系统真正控制的不是 prompt，而是一个持续收敛中的 design state
Persistent State Object

已锁定 DNA

  • 已建立的主题逻辑
  • 稳定的材质 / 表面倾向
  • 当前所处的语义路线
  • 受品牌边界约束的限制条件

可变字段

  • 构图开放度
  • 密度变化
  • 簇状分布方式
  • 运动路径
  • 线索强化强度
核心判断
系统管理的不是 prompt，而是一个逐步收敛的设计状态。
prompt 只是执行封装。真正被解释、被比较、被继承、被调度的对象，是这个状态本身。
模块 04 Agent 编排

编排决策层

决定这一轮应该做哪一种推进动作，而不是直接替代语义判断
Orchestration Logic
Decision A
阶段识别
判断当前处于探索、细化、路线切换，还是收敛阶段。
Decision B
策略选择
决定是扩大试探、稳定当前方向、强化线索，还是缩窄变量空间。
Decision C
控制模式选择
采用 compiler-led、reference-anchored，或 hybrid 控制方式。
Decision D
下一轮实验框架
定义 A / B / C 三个变体各自测试什么变量，避免差异失控。
Agent 的职责不是成为设计逻辑来源，而是决定语义系统下一步应该如何移动。
模块 05 受控执行

执行层

把策略翻译成下一轮可比较、可解释的输出
Controlled Execution
Execution 1
控制指令组装
把当前状态编译成可执行的 prompt / control instructions。
Execution 2
品牌线索注入
只注入与当前方向一致的 brand-aligned semantic cues。
Execution 3
生成三组受控变体
输出 3 个边界清晰的替代方案，而不是随机发散。
Execution 4
结构化对比说明
清楚标记每个方向的差异来源与所测试变量。
输出不是随机 alternatives，而是在锁定 DNA 与可变字段框架内被有意组织的下一轮动作。
状态更新 → 重新解释 → 持续收敛
Feedback Loop
用户评审与偏好信号
用户会选择、拒绝，或部分偏好某个方向。该反馈不会把系统带回空白起点，而是直接回写到下一轮的语义输入与状态更新中，形成持续收敛的多轮循环。
这里被管理的不是 prompt，而是一个在多轮中逐步成形、逐步被收束的 design state。
只有当 semantic layer 已经存在，这种工作流才成立。因为只有当用户反馈、已选图像、品牌线索都能被翻译进同一个可比较的状态空间时，agent 才能真正承担“编排下一步动作”的职责，而不是退回到一次次重写 prompt 的低层循环。
