## 项目动机

这个项目从一开始就不是一个“为了完成某个功能而做的语言学习工具”。它背后的出发点，是把语言学习理解成一种更深层的存在体验。

对作者来说，语言学习一直是 “Study for the sake of itself”。读到《The Language Instinct》时，“语言天赋是一种基因层面的能力” 这个观点，把过去几年零散的体验串成了一条线。学习语言的兴奋，不只是掌握一个单词或一个用法，而像是在唤醒身体里某个古老的本能。这个过程本身，变成了对自身存在的一种觉察。

一些真实的语言时刻包括：

- 西语还只有 A1 水平，但已经能在弗拉明戈课堂里用最基础的词汇和外教交流
- 第一次在音乐里听懂 “ahora”，在舞步中理解 “izquierda”，那种瞬间连接现实的感觉非常强
- 当老师夸 “tu falda bonita”，当在真实场景里逐一解锁 `bastón`、`mantón` 这些词时，能明确感觉到一个世界在展开

这个项目想表达的是：多语言不必被理解成一段沉重而遥远的旅程。所有微小但真实的时刻，都是学习的一部分，也都值得被记录、被回看、被重新激活。

如果现在有人问为什么要学德语、西语，甚至未来还想继续学更多语言，也许很难给出一个功利的答案。更接近真实的说法是：不断创造机会，让自己更靠近语言，靠近文化、声音，以及现实生活里一个又一个细小场景。语言学习本身，成了理解世界、理解自己的方式。

AI，尤其是大语言模型和 Agent 的出现，进一步改变了这种学习方式。它让人开始用 Agent 记录生活、交换记忆、优化表达，让语言学习的阻力被一点点消除，变成更自由、更定制化、也更贴近自我的体验。

## 产品愿景

Language Diary 不是一个把聊天、背词和做题拼在一起的语言学习工具，而是一套围绕日常 ritual（仪式化节律）组织的多智能体学习系统。

它不把语言学习理解为“做题、背词、纠错”的直线流程，而是把它设计成一个贯穿日常生活的认知循环：用户在真实生活中表达，系统接住表达，而不是立刻打断纠错；系统把这些表达转化成可复用表达、知识卡、记忆锚点、日记与测验；然后在未来合适的时间，再温柔地把这些内容唤醒。

一句话概括，这个产品是：

一个基于日常 ritual 的多智能体语言陪伴系统，把用户真实生活中的表达，转化为可记忆、可复用、可回顾的长期学习资产。

## 产品叙事与用户旅程

这个产品把“表达”视为长期资产。它让用户在白天通过真实对话暴露自己的语言边界，在夜间把这些边界转化为可复用的表达记忆，并把语言学习重新嵌入生活的节律之中。

在作品叙事里，作者用一个叫 Mira 的女孩来讲这个产品。

对 Mira 来说，她不是在完成练习，而是在一点点学会：如何用另一种语言，更准确地成为自己。

### 晨间唤醒

清晨，Mira 醒来，脑子还没完全进入状态。她并不想一大早就面对“你昨天学的第 6 个单词还记得吗”这类任务型问题。

她打开应用时，迎面而来的不是任务感，而是一句很轻的开场：“Lass uns eine Erinnerung kurz aufwecken.” 让我们轻轻叫醒一个记忆。

一天不是从输入知识开始，而是从一个熟悉的位置开始开口。有时是一个场景，有时是一个偏好，有时是一个方向，有时是一个未完成的念头。

### 白天陪伴

白天，Mira 并不是在“上课”。她只是像平常一样，和 app 里的 agent 伙伴聊天。她会说今天在学校发生的小事，说自己为什么有点难过，说某个时刻明明有感觉，却不知道怎样用目标语言把它说得更准确。

这个陪伴她的“语言搭子”不是一个不停纠错的老师，更像一个贴身的语言陪伴者。它会顺着 Mira 当下的话题往下走，保持轻量、自然、像朋友一样的追问；它尽量不讲大道理，也不把对话拉回教材。

它更重要的一项工作，是在 Mira 的真实表达里悄悄识别那些“值得升级”的部分：某个句子可以更自然，某个词还不够贴近她真正的感受，某种表达方式其实已经带有她稳定的个性轮廓。

### 学习在悄悄积累

白天的对话不会只停留在聊天记录中。系统会把 Mira 当天说过的话、出现过的表达、关联的词和片段，逐渐组织成可回收的素材：

- 有些进入词汇或知识卡片
- 有些形成表达升级线索
- 有些作为带有生活场景、情绪温度和个人痕迹的事件锚点被记住

这意味着语言不再是抽象规则，而是被绑定到她自己的生活片段上：早餐前的匆忙、和朋友闹别扭后的沉默、某个开心得想立刻告诉别人的时刻。语言从一开始就和记忆缠在一起。

### 晚间回顾

到了晚上，这个 app 的角色会发生变化。白天陪 Mira 说话的搭子，会在夜里变成一个更有整理能力的语言老师。

他不会像传统复习那样只问她今晚要背哪个词，而是更像把 Mira 白天散落的表达重新捧回来：今天你最想重新看一眼的是哪个词、哪个句子、哪一个你没说好的瞬间？

这里最重要的不是复述知识点，而是帮助 Mira 完成一次从“生活中的含混感受”到“可以被再次使用的语言形式”的转化。

### 记忆收回

系统会帮助 Mira 回看一个词在今天真实语境里的意思。它可能会比较相邻表达之间的差别，给出更自然的句子升级，或者只提炼一个她明天还能继续使用的小句型。

语言知识不再以条目的形式出现，而是以“今天我本来想这样说，下次我可以更自然地这样说”的方式留下来。

与此同时，这套系统也不仅仅面向语言表层。它不是心理分析，也不是给 Mira 贴上固定标签，而是更轻地反映：她今天说话时是不是特别谨慎？是不是总在克制、绕开，或者特别努力地让自己的表达显得不冒犯别人？

在这个设想里，语言学习从来不是脱离主体的。一个人怎样说话，往往也暴露了她怎样面对世界。到了晚上，系统不仅帮助 Mira 记住“怎么说”，也帮助她记住“原来我今天是这样在感受、这样在组织自己”。

## 产品的核心认知框架

如果只把它看成一个会说话的语言学习助手，就会低估它真正的系统深度。它更像一套被拆分成多个器官的人工认知系统。

它可以被理解成一条认知循环：

真实生活表达  
→ 捕获与承接  
→ 短时上下文维持  
→ 表达升级  
→ 知识蒸馏  
→ 事件 / 事实记忆编码  
→ 夜间整合  
→ 晨间唤醒  
→ 再次表达与复用

从认知器官的角度看，这套系统包含：

- 感官输入层：DaytimeCaptureAgent
- 工作记忆：SessionStore、MemoryManager
- 海马体 / 事件编码：EventAnchorService
- 事实记忆形成：AutoMemoryCaptureService
- 表达重构系统：AcquisitionAgent、LadderingEngine
- 语义压缩层：KnowledgeAgent
- 叙事沉淀层：DiaryAgent
- 晨间唤醒与回忆层：MorningWakeAgent、MorningRecallAgent

## 系统架构：Ritual Flow

系统整体由两层构成：

- 上层是可见的 ritual flow，也就是 morning、daytime、night 三个节律化阶段
- 下层是支撑这些 ritual 的 orchestrator、memory、agent coordination 和 runtime system

这套系统是一个 stage-based architecture。每个 ritual moment 都有自己的行为目标、记忆表面和编排逻辑。

Orchestrator 是系统的中央调度器。它根据当前 ritual stage 做主路径判断，决定：

- 当前阶段是谁主导
- 什么时候只是回复一条消息
- 什么时候要顺带生成 acquisition card
- 什么时候要顺带生成 knowledge insight
- 什么时候要进入 night wrap-up 或 morning recall

运行时主入口负责：

- 启动 Express
- 初始化 agents、services、repos
- 注册 API 路由
- 提供静态资源
- 启动 backup 和 daily reset 任务

## 多 Agent 系统：角色、Prompt 与交接

Language Diary 不是一个单 prompt 系统，也不是一个页面里塞了很多 tab 的助手。它是一个具备明确认知角色、行为合同与技术交接点的多 agent 系统。

### DaytimeCaptureAgent

DaytimeCaptureAgent 是白天主陪伴 agent，也是主对话入口人格。它的核心目标是：让用户在真实生活语境中放心表达、持续表达、自然表达。

它的任务不是主导 conversation，而是尽可能贴近用户当下的真实表达：

- 回应用户当前消息，而不是抢占话题
- 引用具体细节，维持真实表达的连续性
- 在低干预陪伴中识别值得升级的表达机会

它的 prompt skeleton 强调：

- 必须主要回应最新用户消息
- 不主动开启新话题，除非用户明确切换
- 回答自然、简短，通常 1 到 2 句
- 最多一个问号
- 必须引用一个具体细节体现 active listening
- 不说教、不总结、不主动建议
- 不谈论自己

### AcquisitionAgent

AcquisitionAgent 是表达升级 agent。它负责把用户原句重写成更自然、可偷走、可迁移的表达。

它不是简单润色，而是要求：

- 不沿用原句结构
- 不能只是加个引导词然后重复原句
- 必须在结构上重组
- 输出一个 upgraded expression，以及可复用的 stealable phrases

它的意义在于：把用户临时说出来的话，转成未来还能再用的表达模式。

### KnowledgeAgent

KnowledgeAgent 是知识蒸馏 agent。它把真实表达压缩成结构化学习资产。

它的任务不是泛泛讲解，而是根据用户的最新表达，生成一张有明确 mode 的知识卡。它支持不同类型的知识对象化输出，例如语法升级、语气升级、表达升级等。

### EveningReviewAgent

EveningReviewAgent 承担更明确的“整理者”角色。它会把夜间复习组织成一次有限轮次的聚焦互动：围绕一个词、一个表达、一个句子，或者一个今天的重要瞬间，帮助用户做语义澄清、表达比较、句子升级和与实际使用相关的语法整合。

与传统 tutor 不同，它的重点不只是讲解，而是帮助用户从“今天真实试图表达的东西”出发，把临时性的说法转化成可复用的表达模式。

### NightWrapupAgent

NightWrapupAgent 是夜间闭环 agent。它帮助用户收束一天的表达与学习，并进入温和结束状态。

它的职责包括：

- 生成轻量收束
- 生成 quiz items
- 组织 grammar challenge
- 给出 closing

它要让用户感觉到：

- 今天说过的话不是白说的
- 今天有可带走的语言成果
- 今天的学习有结束感

### DiaryAgent

DiaryAgent 是日记沉淀 agent。它把用户与 companion 的对话线程，转写为一篇自然的第一人称日记。

这里有一个非常严格的原则：它的唯一事实来源是会话线程本身。它只能写用户明确说过的内容，不允许脑补事件、人物、地点和情绪。companion 的问题只提供上下文，不应直接写入 diary 内容。

DiaryAgent 的价值在于：把白天零散的表达沉淀成一段更像“私人记录”的语言，让语言学习不只通向复习，也通向自我叙述。

### MorningWakeAgent

MorningWakeAgent 是晨间个性化记忆唤醒 agent。它的目标是基于事件锚点、词汇和知识资产，生成一个自然、轻柔、个性化的晨间问题。

它要做到：

- 从过去一段时间里挑一个最值得被重新想起的点
- 用用户语言中的词或知识作为钩子
- 生成一句像朋友一样的晨间唤醒问题
- 保持回答范围很轻，只需要 1 到 3 句

### MorningRecallAgent

MorningRecallAgent 是晨间轻 recall agent。它希望用户在大约 3 分钟内完成一次轻量唤醒。

它的原则包括：

- 默认两题，最多三题
- 成功率高，避免挫败感
- 像朋友，不像老师
- 不新增学习内容，只重组旧内容

### QuizAgent

QuizAgent 负责把知识资产转成可练、可测、可解释的练习会话。它不仅生成题目，还要解释为什么某个表达在当前场景更自然、为什么另一个答案不如它合适，帮助用户辨认表达差异。

## 多 Agent 机制的意义

这个多 agent 结构的意义，不是技术上的“拆模块”，而是把语言学习中几个通常彼此割裂的过程重新编织成一条连续链路：

- 表达
- 记录
- 复盘
- 回忆

它不是让一个 agent 同时承担所有任务，而是通过不同阶段的 agent 协作，建立一种围绕表达与记忆的流动。

## 四层记忆结构

这个项目的核心目标，不是让用户单次学会某个词，而是建立一种能够嵌入日常生活节律的语言记忆系统。系统将用户在真实对话中的一次表达，逐步转化为可回忆、可复用、可反思的长期资产。

四层记忆结构包括：

- Event Anchor：将表达绑定到具体生活场景
- Memory Fact：提取用户持续性的偏好、目标与反思
- Knowledge：从真实表达中抽取可复用的语言模式
- Diary：将对话沉淀为第一人称自我叙事

### Layer 1: Event Anchor

Event Anchor 用于记录一次表达背后的生活片段。它的目标不是先分析语言本身，而是先为语言建立一个“发生过的场景坐标”。

记录范围包括：

- 日期
- 时间范围：morning / afternoon / evening / night
- 事件类型：work / food / exercise / social / travel / emotion / routine / reflection / other
- summary / tags / entities（people / places / objects / activities）
- emotionTags / sourceRefs / salienceScore / reusabilityScore
- linkedKnowledgeIds / linkedWordIds / linkedGrammarPointIds
- recallCount / lastUsedAt

这一层让语言学习和生活片段发生绑定。未来回忆的不是抽象表达本身，而是“那天那个时刻，我试图这样表达”。

### Layer 2: Memory Fact

Memory Fact 用于从一次表达中提取更稳定的人物线索。如果 Event Anchor 记录的是“发生了什么”，那么 Memory Fact 记录的是“这个人反复在意什么”。

记录范围包括：

- type：event / preference / goal / reflection
- summary / evidence / tags / entities / emotionTags
- timeScope / noveltyScore / importanceScore / retrievabilityScore
- linkedKnowledgeIds / linkedWordIds / linkedGrammarIds
- status：candidate / confirmed / archived

这一层使系统具备“连续理解用户”的能力。它帮助系统逐渐形成对用户稳定偏好、长期目标和表达倾向的认识，而不只是停留在单次对话。

### Layer 3: Knowledge

Knowledge 层将用户真实表达中的语言价值提炼为可复用的学习对象。它的重点不在离散知识点本身，而在于把“今天说过的话”转化成“明天还能继续用的表达模式”。

Knowledge Card 的记录范围包括：

- mode 分类：语法升级 / 语气升级 / 表达升级
- points 抽取
- span / better / microPattern
- application scenarios
- quiz / recall 使用

这一层承担语言教学功能，但并不是脱离上下文的抽象教学。它始终要回到用户原本的生活场景：这句话为什么当时不好说，怎样升级后更自然，哪些表达可以迁移到未来。

### Layer 4: Diary

Diary 层负责将用户一天中的零散表达整理为第一人称叙事。它更接近用户的主观感受与自我记录。

Diary 基于用户在会话中明确说过的话生成，并可保存、编辑、再次读取。

这一层的目标不是教学，而是让用户重新拥有自己的表达。它让语言学习不只留下“学到了什么”，也留下“我这一天是怎样被说出来的”。

## End-to-End Flow

一条用户表达进入系统后的完整路径可以概括为：

- 用户在白天对话中说出一句真实的话
- 系统生成 agent 回复
- 系统尝试将这句话写入 Event Anchor
- 若处于 daytime-capture 阶段，系统进一步评估是否生成 Memory Fact
- 系统可将相关语言内容提炼为 Knowledge
- 晚间或事后，系统将当天对话整理为 Diary
- 保存 Diary 后，Diary 本身又可以再次生成新的 Event Anchor

四层共同构成了一条从“即时表达”到“长期资产”的路径：用户说过的话，被记住、被理解、被提炼，也被重新写回她自己的语言里。

## 隐形产品核心：Memory & Regulation

这个产品真正的壁垒不在聊天表面，而在 memory encoding、retrieval、linking、profile adaptation 和 behavioral regulation。

如果把它理解成“一个会说话的语言学习助手”，就会低估它真正的系统深度。对这套系统来说，真正构成产品壁垒的不是聊天界面，而是 memory architecture。

系统并不满足于：

- 用户说一句
- Agent 回一句
- 对话结束即消失

它更想留下的是：

- 能被未来再次调用的事件记忆
- 能被结构化保存的知识资产
- 能被重新激活的表达模式

### SessionStore

SessionStore 是系统最基础的会话工作记忆容器。它让 morning、daytime、night 三个阶段拥有明确的会话边界。

如果没有它：

- 当前对话没有连续性
- 不同 ritual 阶段会互相污染上下文
- 无法清晰区分 morning-recall、daytime-capture、night-wrapup

### MemoryManager

MemoryManager 是一个轻量的 session-level working memory 模块。它维护：

- 当前 session messages
- summary
- memoryStrength
- daily reset

它的意义在于，让系统不仅“存消息”，还对当前会话的工作记忆状态有更细的管理。

### EventAnchorService

EventAnchorService 是系统的事件记忆编码器。它把用户消息或 diary 中的生活片段，转成结构化的 EventAnchor，供未来 recall、linking 和 morning wake 使用。

EventAnchor 的价值在于，它把生活片段变成：

- 有标签
- 可打分
- 可链接
- 可回忆

### AutoMemoryCaptureService

AutoMemoryCaptureService 是系统的事实记忆抽取器。它从用户消息中自动判断：

- 这是不是值得记住的东西
- 它属于哪种记忆类型
- 它的重要性和可检索性如何

MemoryFact 让系统开始拥有关于用户偏好、目标和持续性想法的结构化认识。

### AnchorLinkService

AnchorLinkService 是系统的记忆连接器。它的职责不是生成内容，而是把一个 EventAnchor 与其他学习资产建立联系：

- word entries
- knowledge cards
- grammar points

它让系统具备：

- 从事件跳到词汇
- 从事件跳到知识卡
- 从事件带出语法点

### Guardrails

Guardrails 是系统生成行为的第二道防线。Prompt 定义理想行为，Guardrails 检查实际输出是否偏离，并在必要时修复。

它要防止的问题包括：

- self-fiction
- multi-focus
- lexicon 违规
- stretch quota 超限

Guardrails 的价值在于：

- 不让 daytime agent 漂成老师或心理咨询师
- 不让输出突然发明自己的生活经验
- 不让输出在细节上破坏信任感
- 不让一个好 prompt 因偶发失真而破坏整体产品调性

### LadderingEngine

LadderingEngine 是系统的表达爬梯器。它负责识别用户当前表达的意图，并基于语言与级别，给出更高阶或更自然的表达模板替换。

它的意义是：

- 不必每次显式纠错
- 可以在自然交互中逐步带用户上台阶
- 升级的是 pattern，而不仅是单词

### LanguageProfileManager

LanguageProfileManager 是系统的用户语言画像管理器。它维护：

- 用户 CEFR level
- confidence
- 各项语言技能分数
- generation policy
- domain 配置
- laddering 偏好
- language observations

虽然 profile 驱动还没有完全贯穿所有主链路，但它已经提供了非常关键的产品方向：系统不再只知道“这个人学哪门语言”，而开始知道“这个人当前适合被怎样对待”。

### GenerationPolicyInjector

GenerationPolicyInjector 的作用，是把 profile 和当前场景转成生成行为上的约束，例如：

- maxWordsPerReply
- stretchQuota
- register
- cefrLevel

这让系统不再只是“语言正确地回答”，而是进一步控制：

- 用多长回答
- 用什么 register 回答
- 允许多高级的词
- 在当前 domain 是否可以稍微拉高难度

这是从“智能聊天”走向“有 pedagogical control 的学习系统”的关键一步。

## 当前的系统评估

当前项目已经体现出一套基础的系统执行质量控制框架。整体体验被拆分为 `morning-recall`、`daytime-capture` 和 `night-wrapup` 三个阶段，并通过 orchestrator 与 session store 按日期和阶段分别管理上下文，保证晨间唤醒、白天自由对话和夜间整理之间的体验边界清晰。

多 agent 的职责也相对稳定：白天负责表达捕捉，早晨负责召回，晚上负责整理与教学，日记与知识模块分别承担叙事沉淀和知识对象化，形成了比较明确的协作结构。

### 已有的执行质量控制

在执行质量上，系统已经具备一些直接可见的约束机制。多轮对话并不是简单堆叠历史，而是通过相关性筛选、历史裁剪、重复微问题去重和记忆注入门控来控制上下文质量，避免 agent 偏题、重复追问或被长记忆带偏。

现有控制包括：

- stage 切分保证 morning / daytime / night 的体验边界清晰
- orchestrator 与 session store 按日期和阶段管理上下文
- 相关性筛选、历史裁剪与去重机制降低上下文污染
- 记忆注入门控避免长记忆把当下对话带偏

### 资产如何形成闭环

记忆机制不仅停留在存储层。Event Anchor 会被真实写入、合并、排序，并在 Morning Wake 中再次被选择和使用；使用后还会更新 `recallCount` 和 `lastUsedAt`，说明记忆已经形成了写入、召回、回写的闭环。

Knowledge 资产同样不是一次性输出，它会被保存为结构化卡片，并在 quiz、recall、night wrapup 等后续阶段再次消费。

Diary 则把白天对话整理成第一人称叙事，并在保存后重新回流为 event anchor，使结构化记忆和叙事记忆形成双向联动。

### 异常处理和 Fallback

异常处理和 fallback 机制在当前项目中也有明确体现。

- DaytimeCaptureAgent 在主生成失败时会逐层退化到最小可用回复
- MorningWakeAgent 在 anchor 或生成链路不可用时会回退到确定性问题或通用问题，并显式标记 `fallbackUsed`
- DiaryAgent 在生成失败时会退回基于用户原话拼接的日记，保证输出不为空且不虚构内容
- KnowledgeAgent 的多种 mode 也都具备 fallback card

整体上，系统已经具备“关键链路不断、异常时降级运行”的基本稳定性。

## Prompt 治理、质量控制与产品化

在这个项目里，生成质量并不是“prompt 写得够不够好”这么简单的问题。真正决定产品稳定性的，是一整套面向行为控制的生成治理系统。

它至少包括：

- prompt 分层
- generation policy
- laddering
- memory gating

这套系统定义的不只是“说什么”，还定义：

- 角色是谁
- 语气应该怎样
- 能不能问问题
- 问什么样的问题
- 回答多长
- 不能进入什么模式

## 风险、评估与路线图

多 Agent 产品如果只看“模型回答得像不像”，很容易误判系统质量。这个项目真正应该评估的是：

- 阶段体验是否顺滑
- agent 分工是否稳定
- 记忆是否真的起作用
- 知识资产是否真的被再利用
- fallback 是否足够稳

当前已经不是功能拼接阶段，而是进入了一个更典型的多 agent 系统阶段：功能链路基本跑通，下一步重点将转向召回稳定性、重复控制、重要性判断和系统级质量评估。

未来复杂化后的主要风险包括：

- 召回不稳定
- 重复调用相似记忆
- 重要记忆被遗漏

在 slide 05 的反思中，这些风险被进一步展开为：

- 随着记忆数量增加，召回可能出现不稳定，导致相关内容没有被准确取回
- 相似事件或相似表达可能在不同层级被重复捕捉和重复调用
- 一些真正重要但表达不显著的记忆，可能在阈值过滤和启发式排序中被遗漏
- 系统级评估会比单点功能实现更重要

## 总结

Language Diary 想做的，不是一个更会聊天的语言学习 app，而是一套把日常表达变成长期学习资产的 ritual-based multi-agent system。

它的核心不是题库，也不是纠错，而是把这些过程重新组织成一条连续的认知链路：

- 白天，真实表达被接住
- 当下的话语被升级成更自然的表达模式
- 生活片段被编码成事件与事实记忆
- 夜晚，语言与叙事被重新整理
- 清晨，旧内容被轻柔地唤醒
- 然后再次回到现实生活中的表达

因此，这个产品真正处理的不是“怎么多学几个词”，而是：一个人怎样在生活里慢慢学会用另一种语言说出自己，并且让这些表达不会随着对话结束而消失。
